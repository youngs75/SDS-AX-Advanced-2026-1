# Deep Agents의 서브에이전트 아키텍처 조사 보고서

## 요약
Deep Agents의 서브에이전트 아키텍처는 **메인 에이전트가 작업을 계획·분해하고, 전문화된 서브에이전트를 위임 호출하여 결과를 다시 통합하는 계층형 구조**입니다. 핵심 가치는 **컨텍스트 격리, 작업 분해, 장기 작업 지원, 병렬 실행 가능성**에 있습니다. [1][2]

## 1. 아키텍처 정의
Deep Agents는 복잡한 작업을 위해 설계된 에이전트 구성 방식으로, **planning and task decomposition**, **context management**, **subagent spawning**, **long-term memory**를 핵심 기능으로 제시합니다. [1]

서브에이전트 패턴에서는 중앙의 메인 에이전트(또는 supervisor)가 전체 목표를 유지하면서, 하위 작업을 서브에이전트에게 맡깁니다. 서브에이전트는 메인 에이전트의 확장 모듈처럼 동작하지만, 실행 컨텍스트는 분리되어 있습니다. [2][3]

## 2. 핵심 구성요소

### 2.1 메인 에이전트 / 오케스트레이터
- 전체 목표를 이해하고 작업을 분해함
- 어떤 서브에이전트를 호출할지 결정함
- 서브에이전트 결과를 통합해 최종 답을 생성함 [2][3]

### 2.2 서브에이전트
- 특정 업무에 특화된 하위 에이전트
- 조사, 요약, 검증, 코드 분석 등 개별 역할에 맞게 설계 가능
- 사용자에게 직접 응답하기보다 메인 에이전트에 결과를 반환함 [2]

### 2.3 작업 분배(Task dispatch)
- 메인 에이전트가 큰 문제를 작은 작업으로 나눔
- 각 작업을 적절한 서브에이전트에 위임함 [1][2]

### 2.4 컨텍스트 관리 / 격리
- 서브에이전트는 메인 에이전트의 전체 대화 맥락을 그대로 공유하지 않도록 설계됨
- 이 방식은 컨텍스트 오염을 줄이고, 각 작업의 독립성을 높임 [1][3]

### 2.5 병렬성
- 독립적인 하위 작업은 여러 서브에이전트를 동시에 실행할 수 있음
- 조사·비교·검증처럼 독립성이 높은 작업에서 특히 유리함 [3][4]

## 3. 실행 흐름
1. 사용자가 복잡한 목표를 입력한다.
2. 메인 에이전트가 전체 작업을 계획하고 분해한다.
3. 필요한 서브에이전트를 선택해 호출한다.
4. 각 서브에이전트는 별도 실행 컨텍스트에서 작업을 수행한다.
5. 서브에이전트가 결과를 메인 에이전트로 반환한다.
6. 메인 에이전트가 결과를 통합하고 후속 작업을 결정한다.
7. 최종 응답을 사용자에게 제공한다. [2][3]

## 4. 장점
- **복잡한 작업 분해에 강함**: 큰 문제를 작은 단위로 나눠 처리 가능 [1]
- **전문화 가능**: 서브에이전트마다 다른 역할, 모델, 프롬프트를 적용 가능 [3]
- **컨텍스트 격리**: 한 에이전트의 잡음이 다른 에이전트에 퍼지는 것을 줄임 [3]
- **병렬 처리**: 독립 하위 작업을 동시에 수행할 수 있음 [4]
- **장기 작업에 적합**: 파일 시스템, 메모리, 계획 기능과 결합해 지속적인 작업 수행에 유리 [1]

## 5. 한계
- **비용 증가**: 메인 에이전트와 서브에이전트 호출이 늘어나 토큰/호출 비용이 커질 수 있음 [3]
- **오케스트레이션 복잡성**: 라우팅, 상태 동기화, 병합 설계가 필요함 [4]
- **지연 증가 가능성**: 병렬화가 항상 빠른 것은 아니며 조정 비용이 존재함 [3]
- **결과 병합 난이도**: 출력 형식이 다르면 통합 품질이 떨어질 수 있음 [4]
- **단순 작업에는 과함**: 복잡한 작업이 아닐 경우 단일 에이전트가 더 적합할 수 있음 [2][1]

## 6. 다른 멀티에이전트 패턴과의 비교

### 6.1 Router 패턴
- 입력을 유형별로 분기해 적절한 에이전트로 보내는 구조
- 초기 분기에 강하지만, 작업 도중의 세부 위임에는 약함 [4]

### 6.2 Handoff 패턴
- 제어권을 다른 에이전트에게 넘기는 방식
- 사용자 대화 주체가 바뀌어야 할 때 유용함 [4]

### 6.3 Parallel fan-out / merge
- 여러 에이전트를 병렬로 돌리고 결과를 합치는 구조
- 독립 작업에는 좋지만, 병합 규칙이 약하면 충돌이 생김 [4]

### 6.4 Skills와의 차이
- Skills는 한 에이전트가 필요 시 특정 전문 기능을 로드하는 경량 방식
- Subagents는 별도 컨텍스트에서 독립 실행되는 점이 더 강함 [1][4]

## 7. 적합한 사용 사례
- 리서치/조사 작업
- 코드베이스 분석
- 긴 문서 요약 및 검토
- 단계가 명확한 멀티스텝 실행 작업
- 비교 분석 보고서 생성 [1][3]

## 8. 구현 시 주의점
- **컨텍스트 격리와 정보 전달 균형**: 너무 많이 주면 오염되고, 너무 적게 주면 품질이 떨어짐
- **실패 전파 제어**: 서브에이전트 하나의 실패가 전체 실패로 이어지지 않도록 설계
- **결과 병합 규칙 명확화**: JSON/스키마 기반 출력이 유리
- **관찰가능성 확보**: 호출 로그, 타임아웃, 재시도, 최종 합성 근거를 추적해야 함 [4]

## 9. 사실 검증 결과 반영
검증 결과, 다음 주장은 대체로 사실로 확인되었다.
- Deep Agents는 planning, context management, subagent spawning, long-term memory를 제공한다. [1]
- 메인 에이전트가 서브에이전트를 위임 호출하는 구조가 문서의 대표 패턴이다. [2][3]
- 서브에이전트는 컨텍스트 격리 하에 동작하며 결과를 메인 에이전트에 반환한다. [3]
- 단순한 작업에는 더 단순한 구조가 적합하다는 문서 취지와 일치한다. [1][2]

다만, “서브에이전트를 tools처럼 호출한다”는 표현은 설명상 유용하지만, 정확하게는 **위임(delegation)** 중심으로 이해하는 것이 더 적절하다. [2][3]

## 결론
Deep Agents의 서브에이전트 아키텍처는 **복잡한 작업을 안전하게 분해하고, 전문화된 하위 작업을 컨텍스트 격리 상태로 수행하게 한 뒤, 메인 에이전트가 결과를 재통합하는 구조**입니다. 리서치, 코드 분석, 장기 문서 작업처럼 **깊고 반복적인 작업**에 특히 적합합니다. 반면 단순한 작업에서는 오히려 과도한 오버헤드가 될 수 있습니다. [1][2][3]
