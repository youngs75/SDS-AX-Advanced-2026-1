최대 성공 max_tokens: 120000

======================================================================
OOM이 발생하지 않은 원인 분석
======================================================================

1. vLLM의 메모리 관리 방식 (PagedAttention)
   - vLLM은 KV Cache를 페이지 단위로 동적 할당하는 PagedAttention을 사용한다.
   - max_tokens를 크게 설정해도 실제 생성되는 토큰 수만큼만 KV Cache가 할당된다.
   - 즉, max_tokens=120000이라도 실제 생성이 7000 토큰에서 stop되면
     7000 토큰 분량의 메모리만 사용한다.

2. 실제 생성 토큰 수가 적었음
   - 모델이 EOS(End of Sequence) 토큰을 자연스럽게 생성하여
     모든 테스트에서 ~5,700~8,700 토큰 사이에서 종료(finish_reason=stop)되었다.
   - max_tokens는 "최대 허용치"일 뿐, 모델이 그만큼 생성한다는 보장이 없다.
   - 따라서 max_tokens를 120K로 설정해도 실제 메모리 사용량은 ~9K 토큰 수준이었다.

3. H100 80GB의 충분한 VRAM
   - Qwen3.5-9B (FP16 기준 ~18GB) + KV Cache를 합산해도
     H100 80GB × 0.95 = 76GB 내에서 충분히 수용 가능하다.
   - 단일 요청에서 실제 9K 토큰의 KV Cache는 수백 MB 수준으로 매우 작다.

4. OOM을 실제로 유발하려면
   - 다수의 동시 요청(concurrent requests)으로 KV Cache를 동시에 대량 할당하거나,
   - 프롬프트 자체를 수만 토큰으로 길게 넣어 input KV Cache를 크게 만들거나,
   - --max-model-len을 GPU 용량 이상으로 설정한 뒤 실제로 해당 길이만큼
     생성을 강제해야 OOM이 발생할 수 있다.

결론: max_tokens는 "예약"이 아니라 "상한"이며, vLLM의 PagedAttention이
실제 사용량만큼만 메모리를 할당하기 때문에 단일 요청에서는 OOM이 발생하기 어렵다.
